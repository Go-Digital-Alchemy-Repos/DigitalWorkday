QUICKBOOKS CLIENT MAPPING + SYNC ARCHITECTURE
NON-DESTRUCTIVE • ADDITIVE • SAFE MAPPING • AUDITABLE • DOCUMENTED

You are working in the DigitalWorkday codebase (React frontend, Node/Express backend, PostgreSQL + Drizzle ORM).

GOAL
Implement a safe QuickBooks Online client/customer mapping and sync architecture so that:
- Digital Workday clients can be linked to QuickBooks customers
- mismatched naming does NOT cause incorrect automatic linking
- admins can manually map, review suggestions, create new QuickBooks customers, and remap safely
- future billing/invoice sync can depend on a trusted customer mapping layer

CRITICAL RULES
- NON-DESTRUCTIVE: do not change existing client behavior or billing behavior.
- Do NOT auto-map clients solely by name.
- Do NOT auto-overwrite existing mappings.
- Keep all changes additive and feature-flagged.
- Mapping must be tenant-scoped.
- QuickBooks sync must be auditable.

PHASE 1 — DATA MODEL (ADD A PROPER MAPPING LAYER)
============================================================

Create table:
quickbooks_customer_mappings

Columns:
- id
- tenant_id (indexed)
- client_id (indexed, FK to local clients)
- quickbooks_customer_id (text, indexed)
- quickbooks_display_name (text nullable)
- mapping_status (enum/text):
    "unmapped"
  | "suggested"
  | "mapped"
  | "sync_error"
  | "archived"
- mapping_method (enum/text):
    "manual"
  | "exact_name"
  | "email_match"
  | "created_from_dw"
  | "imported_from_qbo"
  | "reviewed_suggestion"
- mapping_confidence (decimal nullable)   // 0.00–1.00
- is_locked (boolean default false)       // prevents auto-remap
- last_synced_at (timestamp nullable)
- last_sync_status (text nullable)
- last_sync_error (text nullable)
- created_by_user_id
- updated_by_user_id
- created_at
- updated_at

Constraints:
- unique (tenant_id, client_id)
- unique (tenant_id, quickbooks_customer_id)  // prevent one QB customer mapped to multiple active clients

Indexes:
- (tenant_id, mapping_status)
- (tenant_id, quickbooks_display_name)
- (tenant_id, last_synced_at)

OPTIONAL separate log table:
quickbooks_sync_logs

Columns:
- id
- tenant_id
- entity_type ("client_mapping")
- client_id nullable
- mapping_id nullable
- quickbooks_customer_id nullable
- action ("suggested" | "mapped" | "unmapped" | "created_in_qbo" | "sync_updated" | "sync_failed")
- status ("success" | "error")
- message
- payload_json (jsonb nullable)
- created_by_user_id nullable
- created_at

IMPORTANT:
Do NOT rely only on a quickbooks_customer_id column directly on clients as the primary architecture.
If such a field already exists, keep it for compatibility but make the mapping table the source of truth.

PHASE 3 — MAPPING SUGGESTION ENGINE (ASSISTED, NOT AUTO-LINKING)
============================================================

Create:
server/src/integrations/quickbooks/customers/customerMappingSuggestionService.ts

Function:
getSuggestedQuickBooksMatches({
  tenantId,
  clientId
})

Logic:
Generate candidate matches from QuickBooks customers using weighted signals:
- exact normalized company/display name match
- exact email match
- phone match
- strong string similarity on name
- address similarity (optional)
- active/inactive compatibility

Return:
[
  {
    quickbooksCustomerId,
    displayName,
    confidence,
    reasons: ["Exact normalized name match", "Email match"],
    matchedFields: { name: true, email: false, phone: false }
  }
]

CRITICAL:
- Suggestions only.
- Do NOT auto-link based on this.
- If confidence is high, still require admin confirmation unless explicitly allowed later.
- If mapping is locked, do not generate replacement suggestions unless forced.

PHASE 4 — BACKEND MAPPING WORKFLOW
============================================================

Create:
server/src/integrations/quickbooks/customers/customerMappingService.ts

Functions:
- getClientMapping({ tenantId, clientId })
- listClientMappings({ tenantId, status?, search?, limit?, cursor? })
- linkClientToQuickBooksCustomer({ tenantId, clientId, quickbooksCustomerId, actingUserId, method })
- unlinkClientMapping({ tenantId, clientId, actingUserId })
- lockClientMapping({ tenantId, clientId, actingUserId, locked })
- refreshClientMappingStatus({ tenantId, clientId })

Rules:
1) linkClientToQuickBooksCustomer:
- must fail if quickbooks_customer_id already mapped to another active client in same tenant
- must fail if mapping is locked and user is not allowed
- writes mapping_status = "mapped"
- writes mapping_method
- updates display name and sync timestamps

2) unlinkClientMapping:
- soft reset to mapping_status = "unmapped" or archive old mapping log
- do NOT delete audit history

3) lockClientMapping:
- prevents accidental remapping

4) refreshClientMappingStatus:
- fetches latest QB customer details and updates sync metadata
