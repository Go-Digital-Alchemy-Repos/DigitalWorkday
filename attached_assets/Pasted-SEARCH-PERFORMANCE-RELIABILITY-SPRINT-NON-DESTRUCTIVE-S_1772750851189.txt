SEARCH PERFORMANCE + RELIABILITY SPRINT
NON-DESTRUCTIVE • SQL-FIRST • NO API BREAKS • SAFE ROLLOUT • DOCUMENTED

GOAL
Fix global search slowness and unreliability by:
- eliminating broad tenant fetch + in-memory filtering
- removing N+1 client expansion from search
- replacing expensive comment search patterns
- adding search-oriented indexes
- improving frontend error handling and request behavior
- expanding search scope to match user expectations
- adding observability and perf budgets

DO NOT remove features.
DO NOT break existing permissions or tenant scoping.
DO NOT replace search with Elasticsearch/OpenSearch/etc. in this sprint.
Keep this as a Postgres/Drizzle-first optimization.

Feature flags:
- ENABLE_SEARCH_SQL_FIRST
- ENABLE_SEARCH_UI_ERROR_STATE
- ENABLE_SEARCH_EXPANDED_SCOPE
- ENABLE_SEARCH_PERF_LOGGING

PHASE 2 — SQL-FIRST SEARCH ENGINE (CORE FIX)
============================================================

Replace broad fetch + in-memory filtering with entity-specific SQL search queries.

2.1 Create new service:
server/src/services/search/globalSearchService.ts

Export:
searchTenantEntities({
  tenantId,
  userId,
  query,
  maxResults,
  entityTypes?,   // optional filter
  includeExpandedScope? // behind feature flag
})

2.2 Strategy:
For each entity type, run a dedicated SQL query with:
- tenant scoping in SQL
- permission/visibility filters in SQL where possible
- text match in SQL
- ORDER / ranking in SQL or lightweight post-merge ranking
- LIMIT in SQL before hydration
- lightweight column selection only

Entity queries:
A) Clients
B) Projects
C) Tasks
D) Users
E) Teams
F) Comments

2.3 IMPORTANT:
Do NOT:
- load all tenant rows first
- then filter in JS
- then truncate after filtering

Instead:
- query top N per entity first
- then combine + rank + truncate final result set

2.4 Result shaping:
Return lightweight search result items only:
- id
- entityType
- title/displayText
- subtitle/snippet
- route/link target
- maybe a small badge/meta

Do NOT hydrate heavy relations in search results.

PHASE 3 — REMOVE N+1 FROM CLIENT SEARCH PATH
============================================================

3.1 Search must NOT use getClientsByTenant() if that expands contacts/projects per client.

3.2 Create a lightweight client search query:
- select client id
- company_name
- display_name if exists
- status/stage if useful
- maybe updatedAt
No contacts/projects expansion during search.

3.3 If you already have a batched client list service, do NOT use expanded hydration for search.
Search results must remain light.

============================================================
PHASE 4 — COMMENT SEARCH REWRITE
============================================================

Current problem:
Search builds visibleTaskIds from all fetched tasks, then runs:
comments.taskId IN (...)
AND ilike(comments.body, '%query%')

Replace this.

4.1 Create SQL-based comment search query:
- search comments directly
- join comments → tasks → projects/tenant scoping
- apply tenant + visibility filters in query
- apply text search in SQL
- limit early

4.2 If visibility requires task/project access checks:
- enforce them in the SQL join predicates or a scoped subquery
- do NOT build giant ID arrays in application memory

4.3 Return only:
- comment id
- snippet
- taskId/projectId
- minimal route target

PHASE 5 — SEARCH INDEXES (HIGH IMPACT)
============================================================

Add search-oriented indexes via safe migrations.

5.1 If using Postgres, enable pg_trgm if not already enabled.

5.2 Add trigram indexes (or equivalent text indexes) for:
- clients.company_name
- projects.name
- tasks.title
- comments.body

5.3 If expanded scope is enabled, also add indexes for:
- projects.description
- tasks.description
- client display_name / alternate name fields if they exist
- users.name
- teams.name

IMPORTANT:
- Only add indexes that match actual queried fields.
- Keep migrations safe and additive.
- Document index purpose in migration comments if repo style allows.
